# Thermostat_Logic Module

## 1. Purpose of the module

The **Thermostat_Logic** module implements the thermostat logic for the hybrid HVAC system developed in the course *Sistemas Digitales Programables* (Programmable Digital Systems). Its main function is to act as the ÔÇťlogical brainÔÇŁ between:

- The **temperature and humidity references** (setpoints).
- The **simulated measurements** generated in Simulink.
- The future **real measurements** coming from the ESP32.
- The **real control commands** coming from the ESP32 (MANUAL mode).
- The **simulated control commands** coming from `Control_Simulink` (AUTO mode).
- The **operating mode state** (AUTO/MANUAL) and the systemÔÇÖs **global ON/OFF state**.

This block decides at all times which control signals must be applied to the plant (MiniHVAC_Zone), calculates the basic regulation errors (temperature and humidity), and ensures that the HVAC global enable state (on/off) is correctly enforced.

In **phase 2** of the project, **Thermostat_Logic** was developed and tested **independently** from the rest of the modules, with the goal of:

- Verifying its internal AUTO/MANUAL selection logic.
- Validating correct behavior of the global ON/OFF switch.
- Confirming consistent calculation of temperature and humidity errors.
- Integrating the humidity control branch (`u_humid_sim` Ôćĺ `u_humid_cmd`) symmetrically with the fan and heater.
- Ensuring that the module is ready for future integration with the ESP32 and with the MiniHVAC_Zone physical model.

## 2. Module interface

### 2.1. Inputs (Inports / input signals)

The **Thermostat_Logic** subsystem has the following main inputs:

1. `T_set`  
   - Description: Temperature setpoint.  
   - Expected source: user interface / ESP32 / supervisory logic.  
   - Unit: ┬░C.

2. `RH_set`  
   - Description: Relative humidity setpoint.  
   - Expected source: user interface / ESP32 / supervisory logic.  
   - Unit: %.

3. `T_sim`  
   - Description: Simulated zone temperature generated by the MiniHVAC_Zone model.  
   - Unit: ┬░C.

4. `RH_sim`  
   - Description: Simulated zone relative humidity.  
   - Unit: %.

5. `u_fan_real`  
   - Description: Fan control signal coming from the ESP32 (MANUAL mode).  
   - Source: microcontroller (PWM, digital signal, or equivalent).  
   - Unit: dimensionless (0/1 or 0ÔÇô100%).

6. `u_heater_real`  
   - Description: Heater control signal coming from the ESP32 (MANUAL mode).  
   - Unit: dimensionless (0/1 or 0ÔÇô100%).

7. `u_fan_sim`  
   - Description: Fan control signal generated in Simulink (AUTO mode, e.g., output of a PID).  
   - Source: `Control_Simulink` subsystem.  
   - Unit: dimensionless.

8. `u_heater_sim`  
   - Description: Heater control signal generated in Simulink (AUTO mode).  
   - Source: `Control_Simulink` subsystem.  
   - Unit: dimensionless.

9. `u_humid_sim`  
   - Description: Humidity control signal generated in Simulink (AUTO mode), typically the output of a humidity PID controller (`u_RH_ctrl`).  
   - Source: `u_humid_sim` Outport of the `Control_Simulink` subsystem.  
   - Unit: dimensionless (0ÔÇô1, 0ÔÇô100%, depending on the model definition).

10. `mode_auto_manual`  
    - Description: Operating mode selector.  
    - Convention:  
      - `0` Ôćĺ **MANUAL** mode (real ESP32 signals are used).  
      - `1` Ôćĺ **AUTO** mode (simulated Simulink signals are used).  
    - Type: Boolean / integer signal (0 or 1).

11. `hvac_enable`  
    - Description: Global HVAC system enable signal.  
    - Convention:  
      - `0` Ôćĺ HVAC **off** (all control outputs are forced to 0).  
      - `1` Ôćĺ HVAC **on** (the selected AUTO/MANUAL commands are applied).  
    - Type: Boolean / integer signal (0 or 1).

In addition, two inputs have been defined and **reserved for future integration with the ESP32**:

12. `T_real`  
    - Description: Real temperature measured by the physical sensor connected to the ESP32.  
    - Status in phase 2: declared in the interface and internally terminated (not yet involved in the logic).  
    - Expected use: model-versus-reality comparison, alarms, future control based on real measurements.

13. `RH_real`  
    - Description: Real relative humidity measured by the physical sensor connected to the ESP32.  
    - Status in phase 2: declared in the interface and internally terminated (not yet involved in the logic).  
    - Expected use: model-versus-reality comparison, alarms, future control based on real measurements.

These two signals have been kept as part of the interface to make the moduleÔÇÖs **future extension** easier without modifying the connection structure of the main model.

---

### 2.2. Outputs (Outports / output signals)

The subsystem exposes the following outputs:

1. `u_fan_cmd`  
   - Description: Final fan control command.  
   - Result of applying:  
     - AUTO/MANUAL logic to `u_fan_real` / `u_fan_sim`.  
     - Global enable logic `hvac_enable`.  
   - Destination: MiniHVAC_Zone model and/or output toward the ESP32.

2. `u_heater_cmd`  
   - Description: Final heater control command.  
   - Result of applying the same logic as for the fan.  
   - Destination: MiniHVAC_Zone model and/or output toward the ESP32.

3. `u_humid_cmd`  
   - Description: Final humidity control command.  
   - Result of applying:  
     - AUTO/MANUAL logic to `u_humid_sim` and, if applicable, `u_humid_real`.  
     - Global enable logic `hvac_enable`.  
   - Destination: `u_humid_cmd` input of the MiniHVAC_Zone subsystem and/or future output toward physical humidification/dehumidification actuators.

4. `error_T`  
   - Description: Temperature error between setpoint and simulated value.  
   - Definition:  
     \\[  
     error\_T = T\_{set} - T\_{sim}  
     \\]

5. `error_RH`  
   - Description: Relative humidity error between setpoint and simulated value.  
   - Definition:  
     \\[  
     error\_{RH} = RH\_{set} - RH\_{sim}  
     \\]

These two error signals are intended for later use in:

- Controllers (PID or other strategies) implemented in `Control_Simulink`.  
- Alarm logic (for example, excessive or persistent deviations).  
- Supervision and visualization functions.

## 3. Internal model and implementation

### 3.1. General description of the logic

Internally, **Thermostat_Logic** is structured into three functional blocks:

1. **Calculation of temperature and humidity errors**.  
2. **AUTO/MANUAL mode selector** for fan, heater, and humidity.  
3. **Global HVAC ON/OFF switch** applied to the commands after selection.

The goal of this modular implementation is to allow the block to be tested and debugged independently, ensuring that:

- In MANUAL mode, the outputs follow the commands coming from the ESP32 (when available).  
- In AUTO mode, the outputs follow the commands calculated in Simulink (`Control_Simulink`).  
- When the system is disabled (`hvac_enable = 0`), all command outputs are cancelled.

In these phase 2 tests, the goal is **not yet to evaluate the complete physical response of the HVAC system** (temperatures, humidity levels, response times, etc.), but rather **to verify that the command signals** `u_fan_cmd`, `u_heater_cmd`, and `u_humid_cmd`:

- Follow `u_fan_real` / `u_heater_real` / `u_humid_real` in MANUAL mode (when implemented).  
- Follow `u_fan_sim` / `u_heater_sim` / `u_humid_sim` in AUTO mode.  
- Are correctly cancelled when `hvac_enable = 0`.

This is critical for the rest of the project because:

- If the AUTO/MANUAL selection is not correct, the controllers (in Simulink or on the ESP32) would act on a plant or on hardware that is not receiving the expected commands.  
- The errors `error_T` and `error_RH` are interpreted in the context of the setpoints and of what the system is **trying to do**; therefore, it is important that the control commands sent to MiniHVAC_Zone match the correct operating mode and enable state.

---

### 3.2. Error calculation equations and blocks

Two **Sum** blocks are used in Simulink (configured as `|+-`) to calculate the errors:

- Temperature error:  
  \\[  
  error\_T = T\_{set} - T\_{sim}  
  \\]

- Relative humidity error:  
  \\[  
  error\_{RH} = RH\_{set} - RH\_{sim}  
  \\]

Implementation:

- `+` input of the Sum block: `T_set` (or `RH_set`).  
- `Ôłĺ` input of the Sum block: `T_sim` (or `RH_sim`).  
- Output: connected to the `error_T` and `error_RH` Outports.

These errors are used as inputs to the controllers in the `Control_Simulink` subsystem, so **Control_Simulink no longer needs to receive `T_sim` or `RH_sim` directly**, only the errors.

---

### 3.3. AUTO/MANUAL selector (mode Switches)

For each actuator (fan, heater, and humidity), a Simulink **Switch** block is used with the same design principle.

#### Fan

- Top input (`u1`): `u_fan_sim` (AUTO signal).  
- Bottom input (`u3`): `u_fan_real` (MANUAL signal).  
- Middle input (`u2`): `mode_auto_manual`.

Switch parameters:

- `Threshold = 0.5`  
- `Criteria for passing first input = u2 >= Threshold`

Resulting logic:

- `mode_auto_manual = 0` (MANUAL) Ôćĺ `0 >= 0.5` is false Ôćĺ `u3` is selected Ôćĺ **`u_fan_real`**.  
- `mode_auto_manual = 1` (AUTO) Ôćĺ `1 >= 0.5` is true Ôćĺ `u1` is selected Ôćĺ **`u_fan_sim`**.

The output of this Switch is internally named `u_fan_sel`.

#### Heater

- Top input (`u1`): `u_heater_sim`.  
- Bottom input (`u3`): `u_heater_real`.  
- Middle input (`u2`): `mode_auto_manual`.

Identical parameters:

- `Threshold = 0.5`  
- `Criteria = u2 >= Threshold`

Intermediate output: `u_heater_sel`.

#### Humidity

An equivalent selector is implemented for humidity:

- Top input (`u1`): `u_humid_sim` (AUTO signal, output of the humidity PID).  
- Bottom input (`u3`): `u_humid_real` (MANUAL signal from the ESP32, or 0 in this phase if manual mode has not yet been implemented).  
- Middle input (`u2`): `mode_auto_manual`.

Parameters:

- `Threshold = 0.5`  
- `Criteria = u2 >= Threshold`

Logic:

- `mode_auto_manual = 0` (MANUAL) Ôćĺ `u_humid_real` is selected.  
- `mode_auto_manual = 1` (AUTO) Ôćĺ `u_humid_sim` is selected.

Intermediate output: `u_humid_sel`.

---

### 3.4. Global ON/OFF switch (hvac_enable)

A second switching layer is applied to the already-selected signals (`u_fan_sel`, `u_heater_sel`, `u_humid_sel`) using three additional Switch blocks:

#### Fan

- Top input (`u1`): `u_fan_sel`.  
- Bottom input (`u3`): `Constant 0`.  
- Middle input (`u2`): `hvac_enable`.

Parameters:

- `Threshold = 0.5`  
- `Criteria = u2 >= Threshold`

Behavior:

- `hvac_enable = 0` Ôćĺ output = 0 Ôćĺ fan disabled.  
- `hvac_enable = 1` Ôćĺ output = `u_fan_sel` Ôćĺ fan follows AUTO/MANUAL mode.

The output of this Switch is `u_fan_cmd` (Outport).

#### Heater

- Top input (`u1`): `u_heater_sel`.  
- Bottom input (`u3`): `Constant 0`.  
- Middle input (`u2`): `hvac_enable`.

Equivalent parameters.

The final output is `u_heater_cmd` (Outport).

#### Humidity

- Top input (`u1`): `u_humid_sel`.  
- Bottom input (`u3`): `Constant 0`.  
- Middle input (`u2`): `hvac_enable`.

Equivalent parameters.

The final output is `u_humid_cmd` (Outport).

In this way, the global switch `hvac_enable` uniformly affects all three actuation channels (fan, heater, and humidity).

---

### 3.5. Main parameters

- Switching threshold for mode and enable signals:  
  - `Threshold = 0.5` in all Switch blocks.

- Mode conventions:
  - `mode_auto_manual = 0` Ôćĺ MANUAL (ESP32).  
  - `mode_auto_manual = 1` Ôćĺ AUTO (Simulink).  
  - `hvac_enable = 0` Ôćĺ HVAC off (output commands set to 0).  
  - `hvac_enable = 1` Ôćĺ HVAC on.

## 5. Module-level testing

### 5.1. Objective of the tests

The objective of the phase 2 tests is **to validate the logical behavior of the Thermostat_Logic module independently**, before permanently connecting it:

- To the MiniHVAC_Zone physical model.  
- To the ESP32 and its physical peripherals.

In particular, the tests verify that:

- `u_fan_cmd`, `u_heater_cmd`, and `u_humid_cmd` correctly follow the appropriate signals depending on the mode (MANUAL vs AUTO).  
- `hvac_enable` forces the outputs to zero when the system is disabled.  
- The errors `error_T` and `error_RH` are calculated with the correct sign and consistent values.

These tests **do not yet aim to verify the realistic thermal dynamics** of the system, but only:

> **That the AUTO/MANUAL selection logic and the command-signal enable logic work as expected.**

This is essential to ensure that, in the next phases, when the ESP32 and the physical model are connected, the controllers and alarms act on the correct plant with the correct signals.

### 5.3. Observed behavior and validation

In the simulations carried out for phase 2:

- It was verified that, when switching `mode_auto_manual` between 0 and 1, the outputs `u_fan_cmd`, `u_heater_cmd`, and `u_humid_cmd` correctly change source (real Ôćö simulated).  
- It was confirmed that `hvac_enable = 0` forces the outputs to 0, regardless of the mode and of the values in `u_fan_real`, `u_fan_sim`, `u_heater_real`, `u_heater_sim`, and `u_humid_sim`.  
- The errors `error_T` and `error_RH` show values consistent with the differences between setpoints and simulated measurements.

Conclusion of the module tests:

- The observed behavior matches the expected behavior in all defined test cases.  
- The **Thermostat_Logic** module is considered **validated at the logical level**, including the humidity branch, and ready to be integrated with the rest of the system in later phases.

## 6. Connection with other project modules

### 6.1. Signals received

In the context of the overall HVAC system, **Thermostat_Logic** receives:

- From **interface / user / ESP32** modules:
  - `T_set`, `RH_set` (setpoints).  
  - `u_fan_real`, `u_heater_real` (real hardware commands in MANUAL mode).  
  - `u_humid_real` (future real humidity commands in MANUAL mode).  
  - `mode_auto_manual`, `hvac_enable` (mode selection and enable signal).

- From the **MiniHVAC_Zone** physical-simulation module:
  - `T_sim`, `RH_sim` (simulated zone measurements).

- From **Control_Simulink**:
  - `u_fan_sim`, `u_heater_sim`, `u_humid_sim` (simulated commands in AUTO mode).

- From the **ESP32** (future):
  - `T_real`, `RH_real` (real measurements for supervision and alarms).

### 6.2. Signals delivered

- To **MiniHVAC_Zone**:
  - `u_fan_cmd`, `u_heater_cmd`, `u_humid_cmd` as command inputs to the simulated plant.

- To **control and supervision** modules:
  - `error_T`, `error_RH` for controllers (e.g., PID) and alarm logic.

- To the **ESP32** (in the final integration):
  - `u_fan_cmd`, `u_heater_cmd`, and `u_humid_cmd` may be sent as output commands to the real hardware, or used to compare real versus simulated behavior.

### 6.3. Role within the overall HVAC system

In the overall system, **Thermostat_Logic**:

- Centralizes the selection logic between automatic control (Simulink) and manual control (ESP32) for temperature and humidity.  
- Ensures that the HVAC on/off state is respected at the command-signal level.  
- Provides the reference errors required for controller design and alarm implementation.

Correct operation of this module is key to:

- Avoiding situations where the system believes it is in AUTO but is actually applying MANUAL commands (or vice versa).  
- Ensuring that, when temperature and humidity errors are analyzed, they are interpreted in the context of the commands actually sent to the plant.

## 7. Module conclusions

- **Current module status**:  
  - The **Thermostat_Logic** module is **implemented and validated at the logical level** in Simulink within phase 2 of the project, including the humidity control branch.  
  - It has been developed **independently**, following the criterion of *module-based implementation*.

- **Specific tests performed**:  
  - Unit tests were defined and executed in Simulink for MANUAL, AUTO, and HVAC OFF modes, verifying correct switching and error calculation.  
  - Specific tests were added for the humidity branch in AUTO mode, verifying that `u_humid_cmd` follows `u_humid_sim` when the system is enabled.  
  - These tests ensure that the module performs the expected functions before integration with MiniHVAC_Zone and with the ESP32.

- **Relevance for the integration phase**:  
  - Having an already verified thermostat-logic module (temperature + humidity) makes integration with the physical module (MiniHVAC_Zone) and with the microcontroller (ESP32) more robust and predictable.  
  - The calculation of `error_T` and `error_RH`, together with the AUTO/MANUAL selection logic and the `hvac_enable` switch, provides the basis for:
    - Designing and tuning automatic controllers in `Control_Simulink`.  
    - Implementing alarm and safety logic based on simulated and real measurements.  
    - Extending the project in the future toward a fully hybrid HVAC system (Simulink + ESP32) with combined temperature and humidity control.

In summary, the **Thermostat_Logic** module meets the objectives of **phase 2** (module-based implementation, module-level testing, and technical documentation) and is ready to be used as the central logical control component in the integration and final validation phases of the HVAC system.
