# Control_Simulink Module

## 1. Module Purpose

The **Control_Simulink** module implements the **automatic control logic (AUTO mode)** for the MiniHVAC system within the global `MiniHVAC_Top` model.  

Its main role is to:

- Receive the reference errors calculated in the `Thermostat_Logic` block.
- Generate normalized actuation commands for heating, ventilation, and humidification:
  - `u_heater_sim` (heater)
  - `u_fan_sim` (fan)
  - `u_humid_sim` (humidifier)
- Maintain a clear, modular architecture that makes it possible to:
  - Test the controller **independently** from the rest of the blocks (Phase 2 objective).
  - Later integrate it with the `ESP32_Interface` block for hardware-in-the-loop testing.

At this stage of the project, the module is designed for an HVAC system with **heating + ventilation + humidification** in **winter** mode, assuming one important physical limitation:

> The thermal plant does not include an active cooling actuator, so the zone temperature **cannot drop below the ambient temperature \(T_{amb}\)**.  
> The control strategy focuses on keeping the setpoint above \(T_{amb}\) through heating and ventilation.

As future work, the **model extension** is proposed to include a cooling actuator and provide a complete climate control system (heating + cooling).

## 2. Module Interface

### 2.1. Inputs (Inports / input signals)

| Signal       | Type    | Unit | Description                                                                                  |
|-------------|---------|--------|----------------------------------------------------------------------------------------------|
| `error_T`   | double  | °C     | Temperature error, calculated in `Thermostat_Logic` as `T_set − T_sim`.                  |
| `error_RH`  | double  | %      | Relative humidity error, calculated as `RH_set − RH_sim`.                                |
| `T_sim`     | double  | °C     | Simulated temperature of the HVAC zone, coming from `MiniHVAC_Zone` (used for future logic / diagnostics). |

> Note: `T_sim` is included as an input to support future improvements (deadband, protection logic, etc.). In the current version, `error_T` and `error_RH` are used primarily.

### 2.2. Outputs (Outports / output signals)

| Signal           | Type   | Unit | Range   | Description                                                                                   |
|-----------------|--------|--------|---------|-----------------------------------------------------------------------------------------------|
| `u_heater_sim`  | double | dimless  | [0, 1]  | Normalized heating power command generated in AUTO mode.                        |
| `u_fan_sim`     | double | dimless  | [0, 1]  | Normalized fan speed command, mainly associated with excess temperature. |
| `u_humid_sim`   | double | dimless  | [0, 1]  | Normalized humidifier command generated by the humidity loop.                       |

These `_sim` signals are **“controller-proposed”** signals. The final command applied to the plant (`u_heater_cmd`, `u_fan_cmd`, `u_humid_cmd`) is generated in `Thermostat_Logic` after:

- Selecting AUTO/MANUAL mode.
- Applying the `hvac_enable` enable signal.
- Applying any possible alarm logic.

## 3. Internal Model and Implementation

### 3.1. General Description

Internally, `Control_Simulink` is composed of three main subblocks:

1. `Temp_Controller`  
   - Implements a PID controller (used as a PI controller) on the temperature error `error_T`.
   - Generates an intermediate signal `u_ctrl_T` bounded in the interval \([-1, 1]\).

2. `Actuator_Split`  
   - Based on `u_ctrl_T`, distributes the control effort between:
     - `u_heater_sim` (positive part of the control action).
     - `u_fan_sim` (negative part of the control action, converted to a positive value).

3. `RH_Controller`  
   - Implements a PI controller on the humidity error `error_RH`.
   - Its output is saturated in the interval [0, 1] and delivered as `u_humid_sim`.

This structure provides:

- A **single temperature loop** that globally decides whether the system should heat or promote heat dissipation.
- A separate **relative humidity** loop, which only acts when the measured value is below the setpoint (active dehumidification is not available).

### 3.2. Temperature Control – `Temp_Controller` Subblock

The temperature loop is based on:

\[
error_T(t) = T_{set}(t) - T_{sim}(t)
\]

The `Temp_Controller` subblock implements:

- A **PI** controller in parallel form:
  \[
  u_{\text{PID},T}(t) = K_{p,T}\,error_T(t) + K_{i,T}\int error_T(t)\,dt
  \]
- With the output saturated to:
  \[
  u_{\text{ctrl\_T}} \in [-1, 1]
  \]

The value `u_ctrl_T` is interpreted as follows:

- `u_ctrl_T > 0` ⇒ heat is needed ⇒ **heating should be activated**.
- `u_ctrl_T < 0` ⇒ temperature is too high ⇒ **heat dissipation should be promoted** through ventilation.

The Simulink PID block is configured with:

- Controller type: **PI** (derivative action disabled in this phase).
- Form: **Parallel**.
- Output saturated in [-1, 1].
- Antiwindup enabled using the block’s built-in method (back-calculation/clamping).

The specific values of \(K_{p,T}\) and \(K_{i,T}\) have been tuned empirically during the Phase 2 tests.

### 3.3. Actuator Distribution – `Actuator_Split` Subblock

Based on the temperature PI output `u_ctrl_T`, the `Actuator_Split` subblock generates the heating and ventilation signals:

- For the **heater**:

  - `u_ctrl_T` is taken directly and saturated:
    \[
    u_{heater\_sim} = \text{sat}_{[0,1]}(u_{ctrl\_T})
    \]
  - Implemented using a `Saturation` block with limits [0, 1].

- For the **fan**:

  - The sign of `u_ctrl_T` is inverted:
    \[
    u_{ctrl\_neg} = -u_{ctrl\_T}
    \]
  - It is saturated in [0, 1]:
    \[
    u_{fan\_sim} = \text{sat}_{[0,1]}(u_{ctrl\_neg})
    \]

This implies:

- If `u_ctrl_T > 0`:
  - `u_heater_sim > 0`, `u_fan_sim ≈ 0` ⇒ heating is prioritized.
- If `u_ctrl_T < 0`:
  - `u_heater_sim = 0`, `u_fan_sim > 0` ⇒ ventilation is prioritized to dissipate heat.
- For small errors:
  - Both signals may be close to 0, keeping the zone around the setpoint.

### 3.4. Humidity Control – `RH_Controller` Subblock

The humidity loop is based on:

\[
error_{RH}(t) = RH_{set}(t) - RH_{sim}(t)
\]

The `RH_Controller` subblock implements:

- A **PI** controller in parallel form:
  \[
  u_{\text{PID},RH}(t) = K_{p,RH}\,error_{RH}(t) + K_{i,RH}\int error_{RH}(t)\,dt
  \]

- The output is **explicitly saturated** to [0, 1]:
  \[
  u_{humid\_sim} = \text{sat}_{[0,1]}(u_{\text{PID},RH})
  \]

With this saturation:

- If `RH_set > RH_sim` ⇒ the error is positive ⇒ `u_humid_sim` tends to be > 0 and the humidifier turns on.
- If `RH_set < RH_sim` ⇒ the error is negative ⇒ the output is clipped to 0, because the system does not include a **dehumidification** actuator.

The parameters \(K_{p,RH}\) and \(K_{i,RH}\) are also tuned experimentally.

#### 3.4.1. Physical Limitation of the Humidity Loop and Use Case (Winter, Dry Ambient Conditions)

The humidity dynamics equation used in `MiniHVAC_Zone` has the following form:

\[
\frac{dRH}{dt} = -\frac{1}{\tau}(RH - RH_{amb}) + k_{humid}\,u_{humid\_cmd}
\]

When the humidifier is off \(u_{humid\_cmd} = 0\), the zone relative humidity naturally tends toward the ambient humidity \(RH_{amb}\). The humidifier action makes it possible to increase relative humidity above \(RH_{amb}\), but the model does not include any **dehumidification** actuator that can actively reduce humidity below \(RH_{amb}\).

This implies a physical limitation analogous to the temperature limitation:

- The system can match or exceed \(RH_{amb}\) thanks to the humidifier.
- It cannot sustain a humidity setpoint far below \(RH_{amb}\), because without drying actuators, the dynamics will always tend toward the ambient humidity.

Therefore, the main use case considered in this phase is **winter with dry ambient conditions**, where:

- \(RH_{amb}\) is relatively low.
- The control objective is to **increase** the zone humidity to reach a comfort value \(RH_{set}\) such that \(RH_{set} \ge RH_{amb}\).

In the humidity loop tests, setpoints consistent with this physical limitation have been selected so that the system can reach the reference value or approach it reasonably well.

#### 3.4.2. Future Work: Active Dehumidification

Analogously to the extension proposed for temperature, one possible improvement to the model would be to add a dehumidification actuator `u_dehum_cmd`, with an additional term in the humidity equation:

\[
\frac{dRH}{dt} = -\frac{1}{\tau}(RH - RH_{amb}) + k_{humid}u_{humid\_cmd} - k_{dehum}u_{dehum\_cmd}
\]

This extension would make it possible to:

- Reach and maintain humidity setpoints both above and below \(RH_{amb}\).
- Implement a more complete humidity controller, where the positive part of the control action acts on the humidifier and the negative part acts on the dehumidifier.

> *(This improvement is considered future work, aligned with a possible Phase 3 of the project in which the zone hygrometric conditioning model would be completed.)*


### 3.5. Possible Future Extension: Cooling Actuator

To evolve the model toward a complete HVAC system (heating and cooling), the following is proposed:

- Add a new actuator `u_cool_cmd` and an associated power `P_cool`.
- Modify the temperature dynamics equation of `MiniHVAC_Zone` to:

\[
C\frac{dT}{dt} = P_{heater}u_{heater\_cmd} - P_{cool}u_{cool\_cmd}
- k_{loss}(T - T_{amb}) - k_{fan}u_{fan\_cmd}(T - T_{amb})
\]

- Adapt `Control_Simulink` so that the output `u_ctrl_T` is distributed between the heater and the cooling actuator, while keeping the fan as an auxiliary actuator.

> *(This extension is considered future work for a possible Phase 3 of the project.)*

## 4. Configuration and Execution Environment

### 4.1. Module Type

- Type: **Simulink Subsystem**.
- Top-level model: `MiniHVAC_Top`.
- Location: `Control_Simulink` block connected to `Thermostat_Logic`.

### 4.2. Simulation Model Configuration

In the `MiniHVAC_Top` model, the following configuration has been used:

- **Solver**: `Fixed-step` type.
- **Integration step**: `0.1 s`.
- **Typical simulation time**:
  - Between `200 s` and `500 s` for step response tests.
  - Up to `500–1000 s` for humidity tests.

The internal PID blocks in `Control_Simulink` have been configured with:

- `Sample time = 0` (continuous mode).
- Saturated output as described in section 3.
- Antiwindup enabled.

### 4.3. Relationship with the ESP32 Environment

In this phase (Phase 2), `Control_Simulink` runs in pure simulation without direct ESP32 involvement. However, its design is intended to:

- Serve as the **automatic reference generator** (`u_*_sim`) that will be sent to the `ESP32_Interface` block in later phases.
- Enable hardware-in-the-loop tests where:
  - The ESP32 receives the commands `u_heater_sim`, `u_fan_sim`, `u_humid_sim` and translates them into real PWM/ON-OFF signals.
  - The ESP32 sensors (for example, BME280/DHT) provide `T_real` and `RH_real`, which can be compared with the simulated variables `T_sim` and `RH_sim`.

## 5. Module-Level Tests

### 5.1. Test Objective

Verify the correct operation of the `Control_Simulink` module **independently**, before full integration with:

- `ESP32_Interface`.
- `Alarms_Logic`.

The specific objectives of the tests were:

1. Validate that the temperature loop responds correctly to setpoint changes in AUTO mode.
2. Check behavior in **thermal overshoot** scenarios (initial temperature above the setpoint).
3. Verify that the humidity loop tends to reach the setpoint when it is physically reachable.
4. Confirm that the output signals remain within the [0, 1] ranges and respect the distribution logic (heater vs fan).

### 5.3. Observed Behavior (Qualitative Summary)

- **Case T1 (heating):**
  - Initially, `T_sim ≈ T_amb < T_set`.
  - `error_T > 0` ⇒ the temperature PI increases `u_heater_sim` (reaching close to 1 at the beginning).
  - `u_fan_sim` remains close to 0.
  - The temperature rises smoothly and approaches `T_set` with acceptable tuning (without large oscillations).

- **Case T2 (overshoot):**
  - `T_sim(0) > T_set > T_amb` is set.
  - The initial error is negative ⇒ `u_heater_sim = 0`, `u_fan_sim` approaches 1.
  - The temperature decreases toward values close to `T_set`, physically limited by `T_amb`.
  - Once near the setpoint, the PI reduces `u_fan_sim` and, if necessary, introduces small corrections through `u_heater_sim`.

- **Case T3 (setpoints below T_amb):**
  - When `T_set < T_amb`, the system is observed to be unable to reach the setpoint:
    - The thermal dynamics always tend toward `T_amb`.
    - The fan accelerates the return to `T_amb`, but it never lowers the temperature below that value.
  - This behavior is consistent with the **absence of a cooling actuator** and is documented as a model limitation.

- **Case RH1 (humidity, reachable setpoint):**
  - For `RH_set > RH_sim` and `RH_set ≥ RH_amb`, `u_humid_sim` increases and initially tends to saturate, accelerating the humidity increase in the model.
  - As `RH_sim` approaches the setpoint, the PI output decreases and stabilizes at an intermediate value.
  - This case illustrates the main use in winter with dry ambient conditions, where the goal is to raise indoor humidity to a comfort value.

- **Case RH2 (setpoint below `RH_amb`):**
  - For `RH_set < RH_amb`, the system dynamics naturally tend toward `RH_amb`, even with `u_humid_sim = 0`.
  - The controller cannot eliminate the steady-state error because no dehumidification actuator exists.
  - This behavior is documented as a physical limitation of the humidity model, analogous to the temperature limitation with respect to `T_amb`.

### 5.4. Validation and Tuning Comments

- The test results indicate that the temperature loop is **stable** and meets the objective of keeping the zone around the setpoint when it is above `T_amb`.
- The humidity loop behaves consistently with the simplified model (humidification only).
- Empirical tuning of the \(K_p\) and \(K_i\) parameters has been carried out to improve:
  - Settling time.
  - Absence of oscillations.
  - Reduction of prolonged saturation.

## 6. Connection with Other Project Modules

### 6.1. Received Signals

- `error_T`, `error_RH`: calculated in `Thermostat_Logic` from:
  - `T_set`, `RH_set` (coming from the `ESP32_Interface` block or from internal sources).
  - `T_sim`, `RH_sim` (coming from `MiniHVAC_Zone`).
- `T_sim`: coming from `MiniHVAC_Zone`, available for diagnostics and potential additional logic.

### 6.2. Delivered Signals

- `u_heater_sim`, `u_fan_sim`, `u_humid_sim`:
  - They are sent to `Thermostat_Logic`, which:
    - Selects between AUTO (`*_sim`) and MANUAL (`*_real` from the ESP32).
    - Applies the global enable signal `hvac_enable`.
    - Generates the final commands `u_heater_cmd`, `u_fan_cmd`, `u_humid_cmd` for the `MiniHVAC_Zone` plant.

### 6.3. Role within the Global HVAC System

At the architecture level:

1. `MiniHVAC_Zone` models the physical dynamics (temperature, humidity, flow).
2. `Thermostat_Logic` acts as the **supervisor**:
   - Calculates errors.
   - Manages AUTO/MANUAL modes.
   - Applies enable and alarm logic.
3. `Control_Simulink` provides the **automatic control laws**:
   - Temperature and humidity PID/PI.
   - Heater/fan distribution.
4. `ESP32_Interface` connects the model to the real world:
   - Sensor readings (`T_real`, `RH_real`).
   - User input capture (potentiometers, buttons).
   - Application of commands to physical actuators (real fan, symbolic heating element, etc.).

In this way, `Control_Simulink` is the block that defines the **ideal behavior in AUTO mode**, while `ESP32_Interface` and `Thermostat_Logic` adapt it to the real hardware and ensure safe operation.

## 7. Module Conclusions

- The `Control_Simulink` module has been **developed and tested independently**, meeting the objectives of **Phase 2**:
  - Modular implementation of the temperature and humidity loops.
  - Specific tests in Simulink before integration with other blocks.
  - Clear documentation of the interface and internal behavior.

- The current model represents a **heating + ventilation + humidification** system oriented toward winter-type scenarios.  
  The absence of a cooling actuator explains why the temperature cannot drop below `T_amb`. This limitation is clearly identified and documented.
- Similarly, the humidity model only includes a humidification actuator. Relative humidity tends toward `RH_amb` when the actuator is off and can only be raised above that value; maintaining setpoints below `RH_amb` in a sustained way is not possible. This limitation is also explicitly documented in the module documentation.

- The current state of the module can be considered **validated for reachable setpoints**, making it suitable for integration with:
  - `Thermostat_Logic` (mode selection and enable logic).
  - `ESP32_Interface` (tests with real hardware).
  - `Alarms_Logic` (management of out-of-range conditions).

- During the ESP32 integration phase, `Control_Simulink` will provide the reference behavior in AUTO mode, while the microcontroller will:
  - Physically apply the heating, ventilation, and humidification commands to the actuators.
  - Allow MANUAL mode, where the `_real` commands override the `_sim` references.

- As **future work**, the following is proposed:
  - Add a **cooling actuator** to the `MiniHVAC_Zone` model.
  - Extend `Control_Simulink` to manage heating, cooling, and ventilation in a unified way.
  - Refine automatic parameter tuning (tuning techniques) and further study the interaction with the `Alarms_Logic` block.
